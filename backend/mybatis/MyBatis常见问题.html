            # 为什么mapper接口没有实现类还能调用？

            通常，在进行编码的时候，都是创建一个mapper接口和与之对应的xml文件，就能够在代码中直接进行调用，但是，通过接口进行调用的时候，本质上是调用接口的实现类的方法，但是mapper接口没有实现类，还能够调用，这是为什么呢？

            @Mapper
public interface UserDao {
    User getUserById(@Param("userId") String userId);
}

            <mapper namespace="org.example.dao.UserDao">
    <select id="getUserById" parameterType="string" resultType="org.example.entity.User">
        select * from t_user where user_id = #{userId}
    </select>
</mapper>

            带着这个问题，写了一个小demo

            可以看到，userDao是一个代理对象，本质上是通过jdk动态代理实现的，通过MapperProxy#invoke实现，这个代理对象知道我们的接口的全类名

            在xml文件中，我们都有namespace属性，这个跟接口的全类名相互对应，因此就能够找到这个xml文件，另外，每个方法的方法名和xml文件中的每个语句的id一样，这样，就能够知道要调用哪个sql

            因此xml文件就相当于这个接口的实现类，每个语句对应接口中的方法

            # #和$的区别

            <mapper namespace="org.example.dao.UserDao">
    <select id="getUserById" parameterType="string" resultType="org.example.entity.User">
        select * from t_user where user_id = #{userId}
    </select>
</mapper>

            先来看一下打印的日志，直观感受一下两个语句的区别，两个语句的区别就是把#换成$

            # 这是使用#{}传参
==>  Preparing: select * from t_user where user_id = ?
==> Parameters: 12984(String)
# 这是使用${}传参
==>  Preparing: select * from t_user where user_id = 12984
==> Parameters:

            最直观的感受：

            • 使用#{}进行传参的时候，#{}会被替换成？，并且下面的Parameters有值
            • 使用${}进行传参的时候，${}会被替换成变量的值，并且下面的Parameters没有内容

            对于#，经过层层debug，终于找到了设置参数的的地方：ClientPreparedStatement

            由于我这个参数是string类型的，所以是调用的这个方法进行设置的，点击左侧的接口继承图标，可以找到父接口PreparedStatement

            结论：Mybatis在处理#{}的时候，会先替换成？，然后调用PreparedStatement的set方法来赋值，而${}，只是进行简单的替换

            #{}传参数可以有效的防止sql注入，提高系统的安全性

            既然#{}比${}更加安全，那是不是可以无脑全部都用#{}，那${}又有哪些应用场景？

            • 动态表名
            • 动态字段名
            • 拼接sql片段

            因为如果表名，字段名这些是字符串，如果使用#{}去传参的话， 参数会被添加上单引号''，就会变成：

            select * from 'user' ...

            在进行查询的时候，就会出错了，我们希望的表名是没有单引号的，所以需要用${}，表名也是同样的道理

            另外，上面的sql语句，我们传入的是字符串，但是执行的sql的参数却没有带''，所以，在使用${}，需要单引号，则需要我们自己手动加上

            # 这是使用${}传参
==>  Preparing: select * from t_user where user_id = 12984
==> Parameters: 

# 正确的sql
select * from t_user where user_id = '${userId}'
# 添加单引号后，打印出来的sql
select * from t_user where user_id = '12984'

            # 分步查询，延迟加载（懒加载）原理

            Mybatis支持延迟加载来进行按需查询，如果我们用不到一些数据，就不用查了，在需要这些数据的时候再进行查找，这样能够提高效率，节约内存空间，另外，分布查询将一个多表关联sql拆分成了多个sql，通过这种形式，减少了复杂sql的出现

            [举例]一个对象account中有其他对象的引用user，有时候我们只想用对象account，如果关联查询，那么就都会查出来，这个时候可以考虑分步查询延迟加载，当需要用到user对象的时候再进行查找，不用的时候不查

            @Data
public class Account {
    private Integer id;
    private Integer userId;
    private Integer money;
    private User user;
}

            Mybatis支持association和collection延迟加载，即一对一延迟加载和一对多延迟加载，要开启延迟加载功能，需要在mybatis-config.xml中添加：

            <setting name="lazyLoadingEnabled" value="true"/>

            下面通过association来理解延迟加载的原理

            <mapper namespace="org.example.dao.AccountDao">
    <resultMap id="userAccountMap" type="org.example.entity.Account">
        <id property="id" column="id"></id>
        <result property="userId" column="user_id"></result>
        <result property="money" column="money"></result>
        <association property="user" column="user_id" javaType="org.example.entity.User" select="org.example.dao.UserDao.getUserById"></association>
    </resultMap>

    <select id="findAll" resultMap="userAccountMap">
        select * from account
    </select>
</mapper>

            先来看看当只调用findAll

            List<Account> accountList = accountDao.findAll();

            可以看到，当只调用findAll，控制台只执行了一个sql，但是如果在代码中获取user，则会执行多个sql

            List<Account> accountList = accountDao.findAll();
for(Account account:accountList){
    System.out.println(account.getUser());
}

            延迟加载的实现原理

            首先，通过debug可以看到，我们的account对象其实是一个代理对象，并且是使用Javassist技术创建的，也就是说，当我们调用get方法的时候，其实是通过代理对象去调用的

            然后找到核心代码EnhancedResultObjectProxyImpl#invoke

            public Object invoke(Object enhanced, Method method, Method methodProxy, Object[] args) throws Throwable {
    if (!lazyLoader.isEmpty() && !FINALIZE_METHOD.equals(methodName)) {
        if (aggressive || lazyLoadTriggerMethods.contains(methodName)) {
            lazyLoader.loadAll();
        } else if (PropertyNamer.isSetter(methodName)) {//如果调用的是setter方法
            final String property = PropertyNamer.methodToProperty(methodName);
            lazyLoader.remove(property);
        } else if (PropertyNamer.isGetter(methodName)) {//如果调用的是getter方法
            final String property = PropertyNamer.methodToProperty(methodName);
            if (lazyLoader.hasLoader(property)) {
                lazyLoader.load(property);
            }
        }
    }
    return methodProxy.invoke(enhanced, args);
}

            代理对象是什么时候创建的？

            在DefaultResultSetHandler中，当我们为结果做映射的时候，如果发现有属性是懒加载的，就会创建代理对象

            private Object createResultObject(ResultSetWrapper rsw, ResultMap resultMap, ResultLoaderMap lazyLoader,
                                  String columnPrefix) throws SQLException {
    this.useConstructorMappings = false; // reset previous mapping result
    final List<Class<?>> constructorArgTypes = new ArrayList<>();
    final List<Object> constructorArgs = new ArrayList<>();
    Object resultObject = createResultObject(rsw, resultMap, constructorArgTypes, constructorArgs, columnPrefix);
    if (resultObject != null && !hasTypeHandlerForResultObject(rsw, resultMap.getType())) {
        final List<ResultMapping> propertyMappings = resultMap.getPropertyResultMappings();
        //遍历resultmap映射
        for (ResultMapping propertyMapping : propertyMappings) {
            // 如果属性是懒加载的，就创建代理对象
            if (propertyMapping.getNestedQueryId() != null && propertyMapping.isLazy()) {
                resultObject = configuration.getProxyFactory().createProxy(resultObject, lazyLoader, configuration,
                                                                           objectFactory, constructorArgTypes, constructorArgs);
                break;
            }
        }
    }
    this.useConstructorMappings = resultObject != null && !constructorArgTypes.isEmpty(); 
    return resultObject;
}

            总结：延迟加载的原理就是通过代理技术创建目标对象的代理对象（account），当要使用到懒加载对象(user)的时候，发现为null，就会执行事先定义好的sql（association标签中的select属性），将懒加载对象查询出来，然后代理对象调用set方法进行设置，这样，代理对象内部的懒加载对象就有值了，就能够正常使用了

            # 缓存机制

            • 一级缓存LocalCache

            在应用程序的运行过程中，我们有可能在一次数据库会话中，执行多次查询条件完全相同的sql，mybatis提供了一级缓存来优化这种场景，如果是相同的sql，会优先命中一级缓存，避免直接对数据库进行查询，提高性能

            每个sqlSession中持有executor，每个executor中有一个localCache，当用户发起查询时，mybatis根据当前执行的语句生成MappedStatement，在localCache进行查询，如果缓存命中，直接返回结果，如果缓存没有命中，查询数据库，缓存结果写入到localCache，最后返回结果

            1. 一级缓存的生命周期和sqlSession一致
            2. 一级缓存设计简单，是一个没有容量限制的hashMap
            3. 一级缓存的最大范围是sqlSession内部，有多个sqlSession或者是分布式环境下，数据库的写操作会引起脏数据，建议设定缓存级别为Statement

            MyBatis的一级缓存默认是开启的，先来看看一级缓存的效果：

            InputStream inputStream = Resources.getResourceAsStream("mybatis-config.xml");
SqlSessionFactory sqlSessionFactory = new SqlSessionFactoryBuilder().build(inputStream);
SqlSession sqlSession = sqlSessionFactory.openSession();
UserDao userDao = sqlSession.getMapper(UserDao.class);
//两次相同的查询
userDao.getUserById(12984);
userDao.getUserById(12984);

            在上面的代码中，同一个sqlSession，然后两次相同的查询，对于第一次查询，debug发现查询的数据库

            对于第二次查询，debug发现，会从localCache中获取数据，而不查询数据库

            当从数据库中查询到数据之后，会将数据添加到一级缓存中，下次查找的时候，会直接从缓存中获取

            private <E> List<E> queryFromDatabase(MappedStatement ms, Object parameter, RowBounds rowBounds,
                                      ResultHandler resultHandler, CacheKey key, BoundSql boundSql) throws SQLException {
    
    
    List<E> list;
    //先在localCache中占个位置
    localCache.putObject(key, EXECUTION_PLACEHOLDER);
    try {
        //从数据库中查数据
        list = doQuery(ms, parameter, rowBounds, resultHandler, boundSql);
    } finally {
        //移除localCache中的占位符
        localCache.removeObject(key);
    }
    //放入从数据库中查到的数据
    localCache.putObject(key, list);
    if (ms.getStatementType() == StatementType.CALLABLE) {
        localOutputParameterCache.putObject(key, parameter);
    }
    return list;
}

            怎么判断是两次完全相同的查询?什么情况下会命中一级缓存？

            localCache本质上是hashMap，也就是说，只要key相同，就认为是两次相同的查询，这个key，每次进行查询的时候都会创建

            查看key的创建方法，可以发现，要命中一级缓存中的数据，要满足这些条件：

            1. MappedStatement的id相同，这个也就是写sql的xml文件中，每个标签都要带的id（同一个mapper文件，同一个方法）
            2. 查询数据的起止范围相同
            3. sql语句相同
            4. 参数相同（包括参数类型，参数的值）
            5. 环境相同
            6. 两次查询之间不能执行update，delete，insert方法，数据变动了，缓存当然不能使用，缓存也会被清空
            //删除了一些影响阅读的源码后的createCacheKey方法
public CacheKey createCacheKey(MappedStatement ms, Object parameterObject, RowBounds rowBounds, BoundSql boundSql) {
    CacheKey cacheKey = new CacheKey();
    cacheKey.update(ms.getId());
    cacheKey.update(rowBounds.getOffset());
    cacheKey.update(rowBounds.getLimit());
    cacheKey.update(boundSql.getSql());
    List<ParameterMapping> parameterMappings = boundSql.getParameterMappings();
    for (ParameterMapping parameterMapping : parameterMappings) {
        cacheKey.update(value);
    }
    if (configuration.getEnvironment() != null) {
      cacheKey.update(configuration.getEnvironment().getId());
    }
    return cacheKey;
  }
            • 二级缓存

            一级缓存不支持多个sqlSession之间共享，如果要实现共享，就需要二级缓存，开启二级缓存之后，在进入一级缓存的查询流程前，先在CachingExecutor进行二级缓存的查询

            二级缓存开启后，同一个namespace下的所有操作语句，都影响着同一个cache，即二级缓存被多个sqlSession共享，是一个全局变量

            开启二级缓存后，查询流程：二级缓存---一级缓存---数据库

            [注意]Mybatis的二级缓存在多表查询的时候极大可能会出现脏数据，有设计上的缺陷，另外Mybatis自带的二级缓存是但服务器进行工作的，无法实现分布式缓存，所以在分布式环境下，更推荐直接使用Redis，现在的互联网基本上都是分布式微服务的，所以不做过多了解

            # 拦截器Interceptor

            mybatis有一个强大的功能，在执行过程中，能够拦截方法，执行一些增强逻辑，通过实现Interceptor接口，诞生了很多强大的插件，比如分页插件，动态表名插件，非法sql拦截插件...

            这些插件的实现原理都是一样的，只不过这些插件增强的功能各不相同，现在通过实现一个统计sql执行时间的拦截器，来理解底层的原理

            根据官网的说法，想要自定义插件，只需实现Interceptor接口，并指定想要拦截的方法签名即可

            //这里写拦截哪个方法，拦截的是Executor的query方法
@Intercepts({
        @Signature(type = Executor.class,method = "query",args = {MappedStatement.class, Object.class, RowBounds.class, ResultHandler.class})
})
public class TimeInterceptor implements Interceptor {
    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        long startTime = System.currentTimeMillis();
        Object result = invocation.proceed();
        long endTime = System.currentTimeMillis();
        long timeTotal = endTime-startTime;
        MappedStatement mappedStatement = (MappedStatement) invocation.getArgs()[0];
        String sqlId = mappedStatement.getId();
        System.out.println("执行SQL:"+sqlId+",耗时:"+timeTotal+"ms");
        return result;
    }

    //未做任何重写
    @Override
    public Object plugin(Object target) {
        return Interceptor.super.plugin(target);
    }

    //未做任何重写
    @Override
    public void setProperties(Properties properties) {
        Interceptor.super.setProperties(properties);
    }
}

            //添加在mybatis-config.xml中
<plugins>
    <plugin interceptor="org.example.interceptor.TimeInterceptor"></plugin>
</plugins>

            发现成功打印了执行时间，现在通过debug来了解底层原理

            public static void main(String[] args) throws Exception{
    InputStream inputStream = Resources.getResourceAsStream("mybatis-config.xml");
    //解析配置文件，创建sqlSessionFactory
    SqlSessionFactory sqlSessionFactory = new SqlSessionFactoryBuilder().build(inputStream);
    //打开sqlSession
    SqlSession sqlSession = sqlSessionFactory.openSession();
    UserDao userDao = sqlSession.getMapper(UserDao.class);
    userDao.getUserById(12984);
}

            首先，当mybatis解析mybatis-config.xml的时候，会将所有的插件添加到一个List中

            private void pluginsElement(XNode context) throws Exception {
    if (context != null) {
        for (XNode child : context.getChildren()) {
            //解析插件
            String interceptor = child.getStringAttribute("interceptor");
            Properties properties = child.getChildrenAsProperties();
            //创建插件对象
            Interceptor interceptorInstance = (Interceptor) resolveClass(interceptor).getDeclaredConstructor()
                .newInstance();
            interceptorInstance.setProperties(properties);
            //将插件放到一个list中
            configuration.addInterceptor(interceptorInstance);
        }
    }
}

            当打开sqlSession的时候，会创建Executor，如果有针对executor实现插件，创建出来的executor其实是一个代理对象

            //defaultSqlSessionFactory.openSession()实际上调用的内部的这个方法
private SqlSession openSessionFromDataSource() {
    //创建executor
    final Executor executor = configuration.newExecutor(tx, execType);
}

public Executor newExecutor(Transaction transaction, ExecutorType executorType) {
    //如果有针对executor实现插件，最后都会返回代理对象，从而实现插件的增强功能
    return (Executor) interceptorChain.pluginAll(executor);
}
//创建代理对象
Proxy.newProxyInstance(type.getClassLoader(), interfaces, new Plugin(target, interceptor, signatureMap));

            这样，当executor执行query方法的时候，就会执行代理逻辑，就能够执行自定义拦截器中的方法

            //当executor执行query方法的时候，会被代理
public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
    try {
        Set<Method> methods = signatureMap.get(method.getDeclaringClass());
        //有配置对应的拦截器，会执行拦截器的intercept方法
        if (methods != null && methods.contains(method)) {
            return interceptor.intercept(new Invocation(target, method, args));
        }
        return method.invoke(target, args);
    } catch (Exception e) {
        throw ExceptionUtil.unwrapThrowable(e);
    }
}

            总结：mybatis的拦截器的实现是通过代理技术实现的，当针对目标对象添加了拦截器之后，创建的实际上是代理对象，当执行要拦截的方法时，就会执行增强的代理逻辑
